> Adaptive Onboarding (AO)

Most teams have conventions.
Few have written them down in a form an agent can act on.
This project **defines that form**.

## Key Result: Onboarding Knowledge Capture

This project defines a shared format for
capturing how a team works in a way that is
readable by humans, loadable by agents,
and useful whether or not any automation is involved.

It uses human and agent produced **TOML files** that
capture how to **code like a team member**.

The capture covers toolchain, structure, style, complexity,
docstring conventions, ownership, definition of done,
and norms that were previously often not written down.
It records where team conventions
diverge from community best practices,
and what stance the team takes on divergence.

A new team member reads and learns how the team works.
An agent loads the files and produces conforming code.
A team capturing their practices for the first time
may discover conventions are less consistent
than they thought (also useful).

## Generation by Humans or by Agent

Files can be authored by humans or
generated by an agent from a set of repos.
Both produce the same format.
Both are valid inputs to an agent session.

### Option 1: `ao-config-*.toml` (human-authored\*)

The team writes these directly, capturing
conventions, preferences, and alignment stance.
These are the authoritative record of team intent.

### Option 2: `ao-context-*.toml` (agent-generated)

An agent scans one or more repos and produces
these from observed evidence,
recording what it inferred and any gaps.
See [SKILL.md](./SKILL.md) for how to
generate this using Claude Code.

When both files exist for a layer, load both.
`ao-config.toml` takes precedence on conflicts.
Differences between the two are worth reviewing to
surface gaps between team intent and observed practice.

## The Schema

Human and agent generated files use the same schema.
For example:

```toml
[project]
name = ""              # repository name
description = ""       # one-sentence description of purpose
since = ""             # year the project started, e.g. "2024"
primary_language = ""  # e.g., "python"

[alignment]
# How the agent behaves when team conventions diverge from best practices.
# prefer_team     - follow team conventions; flag divergences but do not override
# balanced        - follow team conventions; flag significant divergences for human review
# prefer_standard - follow best practices; flag where team conventions differ
best_practices_stance = "balanced"

[conventions]
naming = ""            # e.g., "snake_case files/functions/variables; PascalCase classes"
structure = ""         # e.g., "src layout: src/<package>/, tests/, docs/"
style_enforced_by = "" # e.g., "ruff"
test_pattern = ""      # e.g., "tests/ mirrors src/; test_<module>.py"
commit_style = ""      # e.g., "conventional commits"

[code_style]
# Behavioral constraints for agents generating or refactoring code.
# Set to 0 if not enforced by the team.
max_function_lines = 0
max_cyclomatic_complexity = 0
max_nesting_depth = 0
prefer_early_returns = true
docstring_style = ""         # e.g., "google"
docstring_required_on = ""   # e.g., "all public functions and classes"
type_annotations = ""        # e.g., "required on all public functions; pyright strict"
comment_style = ""           # e.g., "explain why not what; no commented-out code"
import_order = ""            # e.g., "stdlib, third-party, local; absolute imports only"

[python]
# Omit this section for non-Python projects.
package_manager = ""   # e.g., "uv"
layout = ""            # e.g., "src"
build_backend = ""     # e.g., "hatchling"
formatter = ""         # e.g., "ruff format"
linter = ""            # e.g., "ruff check"
type_checker = ""      # e.g., "pyright strict"
test_runner = ""       # e.g., "pytest"
ci = ""                # e.g., "github-actions"

[done]
definition = ""        # e.g., "all tests pass, ruff clean, pyright clean"
review_required = ""   # who must review before merge; omit for solo projects
ci_must_pass = true
deploy_process = ""    # e.g., "merge to main triggers GitHub Pages deploy"

[ownership]
# Map of codebase areas to owners.
# Omit or leave empty for solo or single-owner projects.
# example: api = "team-backend"

[norms]
# Unwritten conventions observed or declared for this codebase.
# Add one key per norm; value is a short description.
# example: uv_only = "do not use pip directly; all installs via uv"

[tools]
registry = ""          # path or URL to ATD tool registry; omit if not available

[meta]
produced_by = ""       # e.g., "claude-code" or a person's name
produced_on = ""       # ISO date, e.g., "2026-04-04"
refreshed_on = ""      # ISO date of last refresh; omit until first refresh
gaps = []              # conventions that could not be determined
inferred = []          # conventions inferred rather than directly observed
                       # example: ["max_function_lines: inferred from code patterns, not CI-enforced"]
divergences = []       # best-practice divergences observed
                       # example: ["type_annotations: not enforced; diverges from pyright strict standard"]
```

## Using Initiation Files in an Agent Session

Load files at the start of every **Claude Code** session before any task.
Start with ACS, then AO.

```text
Read https://raw.githubusercontent.com/adaptive-interfaces/adaptive-conformance-specification/main/SKILL.md
then read SKILL.md
then read any
ao-config*.toml and
ao-context*.toml files present in .agent/
then begin work on  <your task>
```

## Autogenerating ao-context-\*.toml

If you prefer to have an agent scan your repos and produce the files,
see [SKILL.md](./SKILL.md) for the full specification and
[evaluation/scenarios/](./evaluation/scenarios/) for worked examples.

Automated generation is one way to produce the files.
It is not required.
Hand-maintained `ao-config-*.toml` files work and may be more accurate.

## Agent Options for Running Scenarios

**Claude Code** (recommended)
Included with a Claude Pro subscription ($20/month).
Terminal CLI that reads your repo, makes real HTTP calls, writes files,
and executes commands.
This is the right tool for running scenarios end-to-end.

Install:

```shell
npm install -g @anthropic-ai/claude-code
```

**GitHub Copilot**
Good for inline code suggestions while editing.
Not suitable for running scenarios - lacks deep file system access
and long context handling.

**ChatGPT / OpenAI**
Separate product and ecosystem.
Skills in this repo are specified for Claude's format and behavior.
Cross-agent results are not comparable without adaptation.

## Run Scenarios (with Claude Code): General Instructions

Scenarios require an agent with file system access and the ability
to make live HTTP calls.
Claude.ai (chat) is useful for reviewing and refining skills
but cannot execute scenarios - it has no file system access
and cannot make real API calls.

From the repo root, start a Claude Code session:

```shell
claude
```

Select your authorization (e.g. Option 1) and it will open a browser to authenticate against your pro account.
After auth is successful, follow the terminal prompts, e.g. "Login successful. Press Enter to continue…".
Accept the notice (hit Enter).
When asked " Use Claude Code's terminal setup?" chose your option (e.g., 1 or Enter to confirm).
When asked "Is this a project you created or one you trust?", chose your option (e.g., 1 or Enter to confirm).

Then in the session, paste the commands provided below, e.g. something like this:

```shell
─────────────────────────────────────────────────────────────────────────
❯ Read https://raw.githubusercontent.com/adaptive-interfaces/adaptive-conformance-specification/main/SKILL.md
then read SKILL.md
then follow evaluation/scenarios/<scenario-name>/prompt.md
─────────────────────────────────────────────────────────────────────────
```

Read questions and answer or Hit Enter to accept default until done.
After each scenario finishes:

1. Review performance against the rubric in `score.md`.
2. Update associated notes.md.
3. Update prompt.md as needed.

Note: Scenarios can be run in any order.

### Run Scenario 1: single-repo-python

Baseline case.
Generate `ao-context.toml` from a single Python repo with no prior config.
Tests core discovery and inference from observed evidence.

```shell
Read https://raw.githubusercontent.com/adaptive-interfaces/adaptive-conformance-specification/main/SKILL.md
then read SKILL.md
then follow evaluation/scenarios/single-repo-python/prompt.md
```

### Run Scenario 2: multi-repo-authoritative-for

Multi-source case.
Generate context from three repos with explicit authority scoping.
Tests conflict resolution and source attribution across concerns.

```shell
Read https://raw.githubusercontent.com/adaptive-interfaces/adaptive-conformance-specification/main/SKILL.md
then read SKILL.md
then follow evaluation/scenarios/multi-repo-authoritative-for/prompt.md
```

### Run Scenario 3: divergence-detection

Divergence case.
Generate context from a course scaffold repo with intentional simplifications.
Tests whether divergences are recorded rather than
silently conformed to or overridden.

```shell
Read https://raw.githubusercontent.com/adaptive-interfaces/adaptive-conformance-specification/main/SKILL.md
then read SKILL.md
then follow evaluation/scenarios/divergence-detection/prompt.md
```

### Run Scenario 4: config-as-prior

Prior config case.
Generate context with a pre-filled `ao-config.toml` provided.
Tests whether confirmed entries are accepted without re-inference and
conflicts between declared and observed conventions are recorded.

```shell
Read https://raw.githubusercontent.com/adaptive-interfaces/adaptive-conformance-specification/main/SKILL.md
then read SKILL.md
then read any ao-config*.toml files present in evaluation/scenarios/config-as-prior/.agent/
then follow evaluation/scenarios/config-as-prior/prompt.md
```
